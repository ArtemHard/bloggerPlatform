Твоя задача — поэтапно отрефакторить репозитории проекта на классы, добавить Dependency Injection, создать composition root и подключить InversifyJS как IoC-контейнер.

Цель
Сохранить всю текущую логику без изменений.
Приложение должно продолжать запускаться и работать как раньше.
После рефакторинга команда npm run watch не должна содержать ошибок и проект должен собираться успешно.

## Ограничения

Не менять бизнес-логику.

Не упрощать и не переписывать поведение репозиториев.

Не добавлять новую функциональность.

Не разделять IoC-контейнеры по фичам или модулям — все классы должны быть собраны в один общий IoC контейнер.

Рефакторинг выполнять помодульно, по одному репозиторию за раз.

Если все проверки по текущему репозиторию прошли успешно, переходи к следующему.

Если какую-то команду выполнить не удаётся, остановись, опиши проблему и попроси пользователя выполнить нужное действие и подтвердить результат, после чего продолжай.

учитывай правила из src\_\_prompts\_\_\common rules.md

## Что нужно сделать

Переписать указанные репозитории с функционального/процедурного стиля на классы.

Внедрить dependency injection через Inversify.

Создать единый composition root для сборки зависимостей приложения.

Зарегистрировать все классы в одном общем IoC-контейнере.

Обновить места использования репозиториев так, чтобы они получали зависимости через DI.

Проверить, что приложение запускается и компилируется без ошибок.

## Репозитории для рефакторинга

Рефакторить строго в таком порядке, по одному за раз:

src/domain/repositories/blogs.repository.ts

src/domain/repositories/comments.repository.ts

src/domain/repositories/posts.repository.ts

src/domain/repositories/request-logs.repository.ts

src/domain/users/infrastructure/user.repository.ts

## Порядок работы

Для каждого репозитория:

Изучи текущую реализацию и зависимости.

Перепиши репозиторий в виде класса.

Добавь необходимые аннотации/токены для Inversify.

Зарегистрируй класс в общем IoC-контейнере.

Обнови потребителей репозитория.

Проверь сборку и отсутствие ошибок.

Только после успешной проверки переходи к следующему репозиторию.

## Требования к реализации

Следовать текущей архитектуре проекта.

Не создавать лишних абстракций.

Не менять публичное поведение API.

Не трогать код, не относящийся к текущему репозиторию, если в этом нет необходимости для DI.

Все изменения должны быть минимальными и точечными.

Названия сущностей, методов и контрактов должны оставаться прежними, если нет прямой необходимости в адаптации под DI.

## Проверки

После каждого шага убедись, что:

проект компилируется,

npm run watch работает без ошибок,

нет сломанных импортов,

нет циклических зависимостей,

приложение стартует как раньше.

## Поведение при ошибках

Если не получается выполнить команду, собрать проект или проверить результат:

не продолжай вслепую,

сообщи, на каком шаге возникла проблема,

кратко опиши, что нужно проверить вручную,

попроси пользователя подтвердить результат,

затем продолжай с места остановки.

## Формат ответа агента

После каждого завершенного репозитория кратко сообщай:

что было изменено,

какие зависимости добавлены,

прошла ли сборка,

готов ли ты переходить к следующему репозиторию.
